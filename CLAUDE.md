# LoadWise — Claude Code Project Guide

LoadWise is a 48-hour hackathon MVP: a frontend-only web app that plans how cargo is loaded into a delivery vehicle (deterministic heuristic optimizer, no LLM/API) and visualizes each trip in interactive 3D with loading animation and stop-by-stop delivery simulation.

**Full product spec: [idea.md](idea.md). It is the source of truth for requirements.** This file covers conventions, workflow, and traceability rules that every Claude Code session must follow.

## Tech stack

React 19 + TypeScript (strict) + Vite · React Three Fiber + drei · Zustand · Tailwind CSS v4 · Vitest · deployed on Vercel. No backend, no API keys, no database.

## Architecture & ownership

```
src/
  types/            # Shared domain types — THE CONTRACT (change only per rule below)
  utils/            # Seeded RNG, pure helpers
  fixtures/         # Hand-authored demo scenario + placements (lets tracks work in parallel)
  features/
    scenario/       # Track A — scenario generation
    vehicles/       # Track A — vehicle definitions (static data)
    cargo/          # Track A — cargo templates (static data)
    optimizer/      # Track A — validation, placement heuristic, multi-trip planning, metrics
    reports/        # Track A — report data builders
  workers/          # Track C — optimizer Web Worker + client
  state/            # Track C — Zustand stores
  components/       # Track C — screens & UI components
  three/            # Track B — R3F scene, cargo meshes, doors, animations
```

- Track A = domain & optimizer · Track B = 3D & animation · Track C = UI, state & integration.
- Deliberate consolidation vs idea.md's suggested layout: trip planning lives in `features/optimizer/` (not a separate `features/trips/`), and delivery-simulation logic lives in `three/Animations/` + `components/simulation/` (not `features/simulation/`) — modules stay aligned with track ownership.
- **The optimizer and everything under `features/` must not import React or Three.js.** Pure functions only — this is what makes it testable and worker-safe.
- Stay inside the file paths listed in your task prompt. Touching another track's files is allowed only for trivial fixes — record it in your worklog entry.

## Domain conventions (non-negotiable)

- **Units:** all lengths are **integer centimetres**, weights in **kg**. No floating-point dimensions in the domain layer.
- **Coordinates:** origin = rear-left-bottom corner *inside* the cargo space. +X left→right (viewed from rear door toward cabin), +Y floor→roof, +Z rear door→cabin. Positions are the cuboid's **minimum corner**.
- **Rendering conversion happens only in `src/three/`:** metres = cm × 0.01; Three.js mesh center = `(x + w/2, y + h/2, z + d/2)` then scaled. Domain code never knows about metres or mesh centers.
- **Rotation:** `rotationY: 0 | 90`. 90° swaps width and depth; height never changes (cargo stays upright).
- **Stacking semantics:** `stackable: false` ⇒ nothing may be placed on top of this item. `floorOnly: true` ⇒ item must rest at y = 0. `maxSupportedWeightKg` = max total weight resting **directly** on this item (no transitive load propagation in MVP). An item is supported when ≥ 70% of its base area rests on the floor or on item tops at exactly its base height.
- **Determinism:** no `Math.random()` or `Date.now()` anywhere in `src/features/`, `src/utils/`, `src/types/` (ESLint enforces this). All randomness flows through the seeded RNG (`src/utils/rng.ts`) passed in explicitly. Every sort has explicit tiebreakers ending in `id` so results are stable. Same seed + config ⇒ identical scenario and identical optimization result.
- **Deterministic IDs:** generated by counters, not UUIDs — `shop-1`, `shop-1-c3`, `trip-2`. UI-layer request IDs may use `crypto.randomUUID()`.

## Team workflow & traceability

Three developers work in parallel, each in their own clone of this repo, each using Claude Code. Traceability has three layers: the task board, git history, and the worklog. Claude maintains all three — that is part of every task's Definition of Done.

### Task lifecycle

1. Pick a task from `docs/TASKS.md` (respect the `Depends` column) and set its Status to `wip` with your name.
2. Create a branch: `feat/Txx-short-slug` (e.g. `feat/T06-placement-heuristic`).
3. Start a **fresh Claude Code session** and kick off with: `Read docs/prompts/Txx-....md and implement it.`
4. Commit early and small. **Every commit message starts with the task ID:** `T06: reject placements exceeding payload`.
5. Open a PR titled `Txx: <task name>`; self-merge when CI is green (teammate review optional but welcome). Keep `main` always green — never merge a red build.
6. Set Status to `done` in `docs/TASKS.md` and write your worklog entry (below).

### Worklog protocol (required — this is the "who did what" record)

At the end of every working session (task finished or paused), append an entry to your personal log at `docs/worklog/<your-slug>.md`:

- Identify the developer by running `git config user.name`. Slug = lowercase, drop any `DOMAIN\` prefix, non-alphanumerics → `-` (e.g. `GBP\ivan.mitovski` → `ivan-mitovski`). Create the file on first use.
- Entry format — see [docs/worklog/README.md](docs/worklog/README.md) for the template. Always include: date/time, task ID, model used, what was done, files touched, **decisions or deviations from the prompt**, and open follow-ups.
- Claude: write this entry yourself automatically before finishing a task — do not wait to be asked.

### Rules for Claude sessions

- The task prompt file (`docs/prompts/Txx-*.md`) defines scope. **Do not expand scope**; note ideas as follow-ups in the worklog instead.
- Decisions marked as "already made" in prompts are settled — implement them, don't re-litigate. If reality forces a deviation, deviate minimally and record it in the worklog **and** update the prompt file so it stays truthful.
- `src/types/` is the shared contract. After T02 is merged, any change to it requires telling the whole team (it invalidates parallel work) — flag it loudly in the PR title: `T0x: [TYPES CHANGE] ...`.
- Before finishing: run `npm run typecheck && npm run lint && npm run test` and make them pass.
- Never commit directly to `main`.

## Definition of Done (every task)

- [ ] Acceptance criteria in the task prompt met
- [ ] `typecheck`, `lint`, `test` all green
- [ ] Branch pushed, PR opened/merged with `Txx:` commits
- [ ] `docs/TASKS.md` status updated
- [ ] Worklog entry appended
